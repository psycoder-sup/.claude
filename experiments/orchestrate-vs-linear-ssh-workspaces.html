Linear held the whole picture

  • Guarded the SSH host-injection vector at all four layers — UI, IPC, value type, channel backstop — including restore.
  • Avoided the repo-subdirectory git-routing bug entirely.
  • Minor: an ssh -f pipe-blocking risk on the status indicator, two guarded force-unwraps, one duplicated builder.

Orchestrate lost coherence at the seams

  • Verified the plan's ghostty config.command "must-verify" against source — a real diligence win.
  • Bug: host-injection guard only at creation, not restore — a hand-edited state file slips past.
  • Bug: a workspace rooted at a repo subdirectory loses remote-git routing — a fault that sits between two agents' files.
03

What parallelism bought — and cost

Once planning is set aside, the two execution models converge. Cost lands within 5% because the fan-out tax (23 agents each paying a cold cache-write) is almost exactly offset by the single context's runaway cache-reads. On time, the waves gave orchestrate a small genuine edge — ~6 minutes off an 80-minute build.

The one thing parallelism couldn't buy back is cross-cutting coherence. Splitting the code across file-disjoint implementers means no single agent sees a concern that spans boundaries — which is exactly where both orchestrate bugs live: the host guard needed on two code paths, the registry root that interacts with git's canonicalization. The linear session, holding it all in one context, simply saw them.

When each one wins

This run was the near-ideal case for linear: a plan already specified to file:line, a feature that fits one context, heavy cross-file coupling. The takeaway isn't "linear beats orchestrate" — it's that with a ready-made plan they're roughly interchangeable, and orchestrate's value lives upstream.

  • Reach for /orchestrate when the plan doesn't exist yet (its decomposition is the value), the work is genuinely wide and independent, or a single context would overflow.
  • Stay linear when the plan is already concrete, the feature fits one window, and the changes are tightly coupled — the seams cost more than the parallelism saves.
